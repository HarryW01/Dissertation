\chapter{Introduction}

In this project, a Python programming environment is created targeted at new programmers. The environment builds upon the idea of helping users learn by removing some of the complexity behind programming, creating a less overwhelming experience. In particular, this project focuses on heavily restricting the expressiveness of Python, then gradually increasing it in line with the user's knowledge. This is done by creating an environment that restricts both language features and built-in functions.

\section{Definitions}

A \textit{restriction} consists of removing syntax from the language; code containing restrictions cannot be run, just as code containing a syntax error cannot. Restrictions come in two forms: 
\begin{itemize}
    \item \textit{Language features} or just \textit{features}: 
    Programming language constructs and data structures (e.g. loops, conditionals, functions, lists).
    \item \textit{Built-in functions} or just \textit{built-ins}: The set of Python built-in functions defined in the Python documentation\footnote{\href{https://docs.python.org/3/library/functions.html}{https://docs.python.org/3/library/functions.html}}.
\end{itemize}

\section{Motivation}

One problem many novice programmers face is how overwhelming programming can feel. The combination of learning many abstract concepts and a completely new syntax for representing ideas leads to difficulty \cite{green_problems_1980,lahtinen_study_2005}. Creating new tools to help them overcome these challenges has been a focus of the computing education field. The idea behind many tools is to create a more rigid programming environment, removing some aspect of complexity from the language \cite{bau_learnable_2017, hermans_hedy_2020, valente_simplifying_2022}.
Students often struggle the most with understanding exactly how a computer takes code and reaches the output \cite{lahtinen_study_2005}. Notional machines help explain why; a notional machine is a simplified mental model of a computer which helps learners think through how a program is executed \cite{sorva_notional_2013}. When users are faced with a large, daunting syntax, there is a mismatch between their notional machine and the full capabilities of the language. Thus, difficulty emerges in explaining behaviour, as the user's concept of how the computer works does not match reality \cite{hermans_hedy_2020}.
Adding restrictions reduces this discrepancy.

Block-based languages, such as Scratch\footnote{\href{https://scratch.mit.edu/}{https://scratch.mit.edu/}} or Blockly\footnote{\href{https://developers.google.com/blockly}{https://developers.google.com/blockly}} have seen success in recent years- as of 2024, Scratch had 140 million users \cite{noauthor_2024_2024}. This begs the question of why a restrictive, non-block-based approach could be useful. First, learners eventually need to transition from block-based to text-based to continue progressing. \citet{kazemitabaar_scaffolding_2023} found that a structured middle ground- as would be provided by this project- helps reduce errors that learners make. Second, having a variety of approaches ensures a range of learners are accounted for. People learn in different ways; some may find the block-based approach too slow to use \cite{weintrop_block_2015}. \citet{weintrop_block_2015} found that some older learners perceived block-based languages to not be `real' programming. These people may prefer a tool based on an established programming language, such as Python. Finally, these approaches are not mutually exclusive. Testing this approach on a text-based language may point towards success when the same restrictive approach is applied to a block-based language.

Finally, the tool uses program visualisation, a concept employed by several learning platforms. Restrictions are incorporated into the design of this tool's visualiser; its primary purpose is to help learners understand how restricted features affect the execution of their program. Thus helping users update their notional machine and better understand execution: a key area of difficulty for learners \cite{lahtinen_study_2005}. \citet{sorva_review_2013} identify an number of reasons visualisations are ignored by educators, such as lack of integration, poor usability, and doubts about effectiveness. By creating a fully integrated and usable visualiser and taking a unique approach to visulisation, the project aims to remove these concerns.
